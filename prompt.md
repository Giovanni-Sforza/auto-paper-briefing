# 需求规格说明书 (SRS)
**项目名称**：个人自动化AI学术追踪与简报系统（Auto-Paper-Briefing）
**文档版本**：V1.0
**目标用户**：科研工作者、算法工程师、研究生及需要持续追踪特定领域前沿文献的个人用户。

## 一、 项目概述
### 1.1 项目背景
科研人员日常需要花费大量时间检索、筛选和阅读最新的学术论文。面对海量的文献，手动判断文章是否值得精读效率低下。本项目旨在通过自动化脚本结合大语言模型（LLM），打造一个“个人学术秘书”，自动完成文献检索、信息提取和简报生成。

### 1.2 核心目标
- **自动化追踪**：实现对指定学术源（arXiv、Google Scholar）特定方向论文的自动抓取。
- **降噪与提纯**：利用大模型提取核心信息，严格限制主观评价，只罗列客观事实。
- **零负担管理**：阅后即焚（不占用本地硬盘空间）、自动排重（不重复阅读）、生成轻量化Web端简报。

---

## 二、 核心功能需求

### 2.1 数据源与检索模块
- **支持的数据源**：
  - **arXiv**：作为主要数据源，通过其官方 API 或爬虫获取开放获取（OA）的 PDF 论文。
  - **Google Scholar**：作为辅助数据源，通过指定的搜索语法获取相关论文链接（*注：由于反爬严格，需作异常处理机制*）。
- **检索规则设置**：允许用户在配置文件中自定义“检索关键词（Query）”、“领域（Category）”以及“每次获取的最大数量（Max Results）”。

### 2.2 查重与过滤模块
- **防重复机制**：系统需在本地维护一个轻量级的数据库或历史记录文件（如 `.json` 或 `.sqlite`）。
- **记录字段**：每次处理一篇文章后，需将该文章的 `标题 (Title)`、`作者 (Authors)` 及 `文章唯一ID` 写入记录。
- **执行逻辑**：每次启动检索后，先比对检索结果与历史记录，若存在则自动跳过，仅处理全新文献。

### 2.3 下载与文件管理模块（阅后即焚）
- **自动下载**：将未处理过的新论文 PDF 文件临时下载到本地指定目录。
- **清理机制**：**强制约束**——每一篇论文在完成“文本解析”与“AI总结”后，系统必须立即自动删除该 PDF 文件，确保脚本运行结束后本地不遗留任何 PDF 实体文件。

### 2.4 AI 阅读与总结模块
- **API对接**：支持调用第三方 AI 厂商（如 OpenAI、阿里云、DeepSeek 等）的 API 接口。
- **Prompt 严格约束（核心需求）**：
  - **预设角度**：必须按照用户提前设计好的维度（如：研究背景、核心方法、实验结论）进行拆解。
  - **客观罗列**：只允许提取和罗列文章中明确提及的观点和事实。
  - **禁止评判**：系统指令需强制禁止大模型对文章进行价值评判、优缺点主观分析或使用感情色彩强烈的词汇。
  - **一次一篇** ：通过循环，每次只要求AI模型处理一篇文章，防止上下文超出模型的最佳上下文窗口。并且明确限制返回的内容。每次返回需要使用中文。

### 2.5 简报生成模块
- **输出格式**：HTML 网页文件（本地生成）。
- **内容呈现**：
  - 文件命名以当日日期为准（如 `Daily_Paper_20231025.html`）。
  - 清单式排版，每篇论文自成一个独立区块。
  - 区块内包含：文章标题、作者列表、AI 提取的客观总结（分点显示）。
  - **原文入口**：每篇文章必须附带一个“原文链接/下载链接”按钮，点击后跳转至原网站，供用户后续自行下载精读。

## 三、 非功能性需求

### 3.1 性能与稳定性
- **并发与限流**：考虑到学术网站和 AI API 的限流政策（Rate Limit），系统应具备合理的请求间隔（Sleep机制）和重试机制（Retry）。
- **大文本处理**：PDF 解析可能产生超长文本，系统需具备截断或分页策略，确保不会超过大模型 API 的单次 Token 上限。

### 3.2 易用性
- **一键运行**：系统需封装为简单的终端命令或一键执行脚本，无需复杂的图形界面。
- **易于配置**：所有的易变参数（API Key、检索关键词、AI总结角度）应提取到一个统一的配置文件（如 `config.yaml` 或 `config.ini`）中。

## 四、 业务流程图（系统执行逻辑）

1. **[启动脚本]** 读取本地配置文件（关键词、API Key、总结角度）。
2. **[加载历史]** 加载本地已处理论文记录（如 `history.json`）。
3. **[执行检索]** 向 arXiv / Google Scholar 发起关键词检索。
4. **[循环处理]** 遍历检索结果：
   - *判断*：是否在历史记录中？ -> 【是】跳过。
   - *判断*：是否在历史记录中？ -> 【否】继续。
   - **[下载文献]** 下载该篇论文的 PDF 到本地临时目录。
   - **[文本提取]** 将 PDF 转换为文本格式。
   - **[AI 分析]** 将文本及预设 Prompt 发送至 AI API，获取客观总结。
   - **[本地清理]** **立即删除**刚刚下载的本地 PDF 文件。
   - **[更新记录]** 将该文章的标题、作者存入内存中的已处理列表。
5. **[生成报告]** 循环结束后，将本次所有 AI 总结打包渲染为 HTML 网页文件并保存至本地。
6. **[持久化]** 将更新后的已处理列表覆写保存到 `history.json`。
7. **[运行结束]**

## 五、 潜在风险与说明（Risk & Mitigation）

| 风险点 | 说明 | 应对策略 / 建议 |
| :--- | :--- | :--- |
| **Google Scholar 反爬** | 谷歌学术对自动化请求的封锁极其严厉，极易触发 CAPTCHA 或封禁 IP。同时很多链接指向付费数据库（知网、IEEE），无法直接下 PDF。 | **建议第一阶段以 arXiv 为主数据源**，因为 arXiv 提供官方开源 API 和纯净的 PDF 下载链接。Google Scholar 可作为后期探索的增强功能。 |
| **大模型幻觉** | 尽管提示词要求“客观不评判”，但大模型偶尔仍可能自行脑补。 | 在 API 调用时，将大模型的 `Temperature` 参数设置为极低（如 0 或 0.1），最大程度限制其发散思维。 |
| **PDF 解析乱码** | 数学公式和双栏排版的 PDF 在提取文本时可能出现乱码，影响大模型阅读。 | 采用业内成熟的 PDF 解析库（如 `PyMuPDF`），并在提取时仅抽取前 N 页（通常摘要和引言已经包含足够信息）。 |
